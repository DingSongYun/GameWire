## Context

GameWire 是一个从零构建的游戏行业 & AI 技术资讯聚合平台。目标用户是游戏公司技术团队，需要从多个异构数据源（资讯网站、社交平台、技术社区、RSS）统一采集内容，经过处理和分类后在 Web 端展示，并支持团队协作。

当前状态：无现有代码库，完全新建。未来需要扩展到桌面终端应用，因此后端必须提供干净的 API 层以便对接。

关键约束：
- 数据源异构性高（HTML 爬取、REST API、RSS/Atom），需要统一的适配器模式
- 内容为中英双语，需要语言检测和翻译能力
- 团队共享场景要求多用户支持和基本的权限控制
- 采集频率需要合理调度，避免被数据源封禁
- 未来需对接桌面客户端，API 设计需前后端解耦

## Goals / Non-Goals

**Goals:**

- 建立可扩展的多源内容采集框架，支持通过插件/适配器快速接入新数据源
- 实现自动化内容处理管线：采集 → 清洗去重 → 摘要 → 分类 → 存储
- 提供现代化的 Web 仪表板，支持浏览、搜索、筛选和趋势可视化
- 支持团队多用户访问与基础协作功能
- 后端 API 设计为前后端分离，为未来桌面端集成打好基础

**Non-Goals:**

- 不做原创内容生产，仅聚合和处理外部资讯
- 不做全文内容存储/镜像（仅存摘要和元数据，链接回原文）
- 不做移动端 App（第一阶段仅 Web，桌面端为后续规划）
- 不做实时推送/即时通知（第一阶段为拉取模式，用户主动刷新查看）
- 不做复杂的权限分级（第一阶段仅区分管理员和普通用户）

## Decisions

### D1: 技术栈选择 — Python 后端 + React 前端

**选择**: 后端使用 **Python (FastAPI)**，前端使用 **React + TypeScript (Vite)**

**理由**:
- Python 在数据采集（Scrapy、BeautifulSoup、feedparser）和 AI/NLP（LangChain、OpenAI SDK、transformers）领域生态最成熟
- FastAPI 提供高性能异步支持和自动 OpenAPI 文档生成，天然适合 API-first 设计
- React + TypeScript 是前端主流选择，组件生态丰富，团队招聘和维护成本低
- Vite 构建速度快，开发体验优秀

**替代方案**:
- Node.js 全栈（优点：统一语言；缺点：AI/NLP 和爬虫生态不如 Python）
- Go 后端（优点：性能好；缺点：AI 生态弱，开发效率低于 Python）

### D2: 数据库 — PostgreSQL + Redis

**选择**: **PostgreSQL** 作为主存储，**Redis** 作为缓存和任务队列

**理由**:
- PostgreSQL 支持全文搜索（tsvector）、JSONB 字段（灵活存储不同源的元数据）、成熟的时序查询
- Redis 用于：采集任务队列、热点内容缓存、API 速率限制计数器
- 两者都有成熟的 Python 驱动和容器化支持

**替代方案**:
- MongoDB（优点：schema-free 适合异构数据；缺点：全文搜索不如 PG，事务支持弱）
- Elasticsearch（优点：搜索强大；缺点：运维成本高，作为主存储过重，可后续引入）

### D3: 内容采集架构 — 适配器模式 + 调度器

**选择**: 采用 **Source Adapter 模式**，每个数据源实现统一接口，由中央调度器按 cron 频率触发

**理由**:
- 各数据源协议差异大（RSS/Atom、REST API、HTML 爬取），适配器模式提供统一抽象
- 调度器基于 **APScheduler** 或 **Celery Beat**，支持每个源独立配置采集频率
- 新增数据源只需编写一个新 Adapter 类，无需修改核心管线

**架构**:
```
SourceAdapter (抽象基类)
├── RSSAdapter          — feedparser 解析 RSS/Atom
├── TwitterAdapter      — Twitter/X API v2
├── RedditAdapter       — Reddit API (PRAW)
├── HackerNewsAdapter   — HN Firebase API
├── WebScraperAdapter   — BeautifulSoup/Scrapy 通用爬虫
└── GitHubAdapter       — GitHub API (Trending)
```

**替代方案**:
- Scrapy 全家桶（优点：爬虫功能全面；缺点：不适合 API 类数据源，框架较重）
- 事件驱动/流式采集（优点：实时性好；缺点：复杂度高，第一阶段不需要）

### D4: AI 处理管线 — LLM API 调用

**选择**: 使用 **OpenAI API (GPT-4o-mini)** 进行摘要生成、分类和翻译，通过 LangChain 编排

**理由**:
- GPT-4o-mini 性价比高，适合大量短文本的摘要和分类任务
- LangChain 提供 prompt 模板管理、输出解析、重试逻辑等便利功能
- 避免本地部署模型带来的 GPU 成本和运维负担

**处理管线**:
```
原始内容 → 语言检测 → 去重(SimHash) → AI摘要 → AI分类/标签 → 翻译(如需) → 入库
```

**替代方案**:
- 本地模型（Llama/Qwen）— 后续可作为降本方案，当前优先快速上线
- 传统 NLP（TF-IDF + 规则分类）— 准确度低，维护成本高

### D5: 趋势分析 — 基于时间窗口的统计聚合

**选择**: 基于 **滑动时间窗口** 的关键词频率统计 + 话题聚类

**理由**:
- 统计每日/周/月的关键词和标签出现频率，识别上升趋势
- 利用 AI 提取的标签做聚合，避免复杂的 NLP 管线
- PostgreSQL 时序查询满足需求，无需引入专门的时序数据库

**替代方案**:
- 实时流处理（Kafka + Flink）— 过度工程，数据量不需要
- 专用趋势分析服务 — 复杂度和成本不匹配

### D6: 认证方案 — JWT + 简单角色

**选择**: **JWT 无状态认证**，两种角色（admin / member）

**理由**:
- 团队规模有限，简单角色模型足够
- JWT 适合 API-first 架构，桌面端对接方便
- 后续可平滑迁移到 OAuth2/SSO

**替代方案**:
- Session 认证（缺点：不适合多端、无状态 API 设计）
- 第三方认证服务如 Auth0（优点：功能全；缺点：外部依赖，团队小场景过重）

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **第三方 API 变更/限流** — Twitter/Reddit 等平台频繁调整 API 政策和费率 | 数据源断供，采集中断 | → 适配器模式隔离变更影响；为每个源配置独立重试和降级策略；优先支持 RSS 作为最稳定的数据源 |
| **AI API 成本失控** — 大量文章的摘要和分类调用可能产生高额费用 | 运营成本超预算 | → 使用 GPT-4o-mini 等低成本模型；设置每日 token 预算上限；对低优先级内容跳过 AI 处理；缓存相似内容的处理结果 |
| **内容去重不准确** — 同一新闻被多个源报道，SimHash 可能误判 | 信息冗余或误删 | → SimHash + URL 去重双重机制；保留去重日志支持人工复核；可调整相似度阈值 |
| **爬虫被封禁** — 高频采集触发目标网站反爬机制 | 数据源不可用 | → 尊重 robots.txt；合理设置采集间隔（每源最低 15 分钟）；使用 User-Agent 标识；优先使用官方 API |
| **单体架构瓶颈** — 第一阶段为单体部署，后续可能遇到扩展问题 | 性能瓶颈 | → 后台采集管线与 API 服务分离部署；数据库连接池管理；后续可拆分为微服务 |

## Migration Plan

此为全新项目，无需迁移。部署策略：

1. **阶段一：开发环境** — Docker Compose 本地运行全部服务（API + DB + Redis + 前端）
2. **阶段二：团队测试** — 部署到单台云服务器，Docker Compose 编排
3. **阶段三：生产部署** — 根据团队规模决定是否上 K8s 或保持 Docker Compose

回滚策略：数据库使用 Alembic 迁移管理，每次部署前备份数据库，支持版本回退。
